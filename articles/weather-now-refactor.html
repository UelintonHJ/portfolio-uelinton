Case Study

Refatorando o Weather Now: de projeto funcional a produto evolutivo

Refatorei uma aplicação em JavaScript puro para reduzir acoplamento, melhorar previsibilidade de estado e elevar a experiência do usuário sem introduzir frameworks.

8 min de leitura

Publicado em Abril de 2026 • por Uelinton Janke

✔ Separação de responsabilidades ✔ Estado previsível ✔ UX sem alert() ✔ PWA confiável
Repositório Deploy

Resumo rápido

  • Centralização de DOM reduziu duplicação
  • Separação de responsabilidades melhorou manutenção
  • Remoção de alert() melhorou UX
  • Service Worker corrigido para uma PWA confiável

Contexto

O Weather Now nasceu como um projeto pessoal com objetivos claros: praticar consumo de API (OpenWeather), construir uma UI moderna com glassmorphism, implementar dark mode, transformar a aplicação em uma Progressive Web App (PWA) e desenvolver tudo utilizando JavaScript puro.

O aplicativo cumpria bem sua proposta. Permitia busca por cidade, exibia o clima em tempo real com ícones animados, era responsivo, instalável como PWA e possuía dark mode. Estava funcional, bonito e publicado.

Mas funcional não é o mesmo que evolutivo.

Com o tempo, ficou evidente que, embora nada estivesse quebrado, a estrutura apresentava fragilidades típicas de um projeto que funciona hoje, mas escala mal amanhã. A função getWeather() concentrava responsabilidades demais. Havia chamadas repetidas de document.getElementById() espalhadas pelo arquivo. O uso de alert() quebrava o fluxo da experiência no mobile. A lógica dos ícones dependia de múltiplos if/else. O Service Worker possuía caminhos inconsistentes. Não existia loading state nem persistência do contexto do usuário.

Nada estava errado. Mas estava frágil. E fragilidade é o oposto de produto.

A refatoração teve um objetivo claro: melhorar a estrutura sem reescrever tudo, sem adicionar frameworks e sem alterar a experiência visual. A meta era tornar o código previsível, reduzir acoplamento, melhorar o feedback ao usuário e preparar o projeto para evolução futura.

Refatorar não é reescrever. É melhorar sem quebrar.

Centralização de DOM

A primeira melhoria foi a centralização das referências de DOM em um único objeto. Em vez de múltiplas chamadas dispersas, passou a existir um ponto de verdade (const DOM = {...}). Isso reduziu o risco de erro, acelerou a leitura e deixou a estrutura da interface clara em poucos segundos, além de abrir caminho para futura componentização.

Antes, o código estava espalhado assim: 

            
                document.getElementById("cityInput")
                document.getElementById("searchBtn")
                document.getElementById("cityName")

Depois da refatoração, passou a existir um único ponto de verdade: 

            
                const DOM = {
                    cityInput: document.getElementById("cityInput"),
                    searchBtn: document.getElementById("searchBtn"),
                    cityName: document.getElementById("cityName"),
                    temperature: document.getElementById("temperature"),
                    description: document.getElementById("description"),
                    weatherResult: document.getElementById("weatherResult"),
                    weatherIcon: document.getElementById("weatherIcon"),
                    themeBtn: document.getElementById("toggleTheme"),
                };

Separação de responsabilidades

Em seguida, houve a separação de responsabilidades. Antes, uma única função validava dados, chamava a API, tratava erros, manipulava o DOM e controlava estado. Depois da refatoração, cada função passou a ter um papel específico: fetchWeather para dados, renderWeather para interface e getWeather para orquestrar o fluxo. O resultado foi um código mais previsível e menos arriscado para mudanças futuras, uma arquitetura mínima viável. 

            
                async function fetchWeather(city) { ... }
                function renderWeather(data) { ... }
                async function getWeather() { ... }

Substituindo alert() por feedback contextual

O alert() foi substituído por feedback contextual dentro da própria interface. O erro passou a aparecer onde o usuário já está interagindo, mantendo a fluidez da experiência e preparando terreno para melhorias futuras de acessibilidade. Uma pequena mudança técnica com grande impacto na percepção de qualidade.

Antes o alert():

  • Quebrava o fluxo
  • Era agressivo no mobile
  • Não combinava com uma UI moderna

Agora, com a substituição por feedback contextual: 

            
                <p id="errorMessage" class="error hidden"></p>

                function showError(message) { ... }
                function clearError() { ... }

O erro aparece no próprio fluxo da interface, sem interrupções desnecessárias.

Ícones declarativos

A lógica dos ícones deixou de depender de múltiplos condicionais e passou a utilizar um mapeamento declarativo. Isso tornou o código mais explícito, escalável e fácil de expandir.

Antes: 

            
                if (condition.includes("clear")) ...
                else if (condition.includes("cloud")) ...

Depois: 

            
                const WEATHER_ICONS = {
                    clear: "clear.json",
                    cloud: "clouds.json",
                    rain: "rain.json",
                    storm: "storm.json",
                    thunder: "storm.json",
                    snow: "snow.json",
                    mist: "clouds.json",
                };
            
        
            
                function loadWeatherIcon(condition) {
                    const key = Object.keys(WEATHER_ICONS)
                        .find(k => condition.includes(k));

                    if (!key) return;

                    DOM.weatherIcon.innerHTML = `
                        <lottie-player
                            src="assets/icons/weatherIcon/${WEATHER_ICONS[key]}"
                            autoplay
                            loop>
                        </lottie-player>
                    `;
                    }

Adicionar novos estados climáticos agora é simples e previsível.

Tema centralizado e previsível

O sistema de tema também foi centralizado em uma função clara (setTheme), com persistência via localStorage. O tema passou a ter contrato definido, permitindo futuras evoluções como tema automático por sistema ou por horário, sem necessidade de reestruturação. 

                
                    function setTheme(theme) {
                        document.body.classList.toggle("dark", theme === "dark");
                        localStorage.setItem("theme", theme);
                    }

Inicialização: 

            
                const savedTheme = localStorage.getItem("theme") || "light";
                setTheme(savedTheme);

Loading state

Foi implementado também um loading state. O botão de busca agora comunica quando está processando, evita múltiplos cliques e reduz a ansiedade do usuário. Pequenos detalhes que elevam a maturidade do produto. 

            
                function setLoading(isLoading) {
                    DOM.searchBtn.disabled = isLoading;
                    DOM.searchBtn.innerText = isLoading ? "Buscando..." : "Buscar";
                }

Aplicando com finally, garantindo consistência e eliminando estados inconsistentes de interface.

Enter para buscar e persistência da última cidade

A experiência foi aprimorada com suporte à tecla Enter para buscar, reaproveitando o fluxo já existente, sem duplicar lógica. Além disso, a aplicação passou a persistir a última cidade pesquisada, restaurando-a automaticamente ao iniciar.

O fluxo foi reaproveitado para respeitar a expectativa natural do usuário: pressionar Enter durante a busca. 

            
                DOM.cityInput.addEventListener("keydown", (event) => {
                    if (event.key === "Enter") {
                        getWeather();
                    }
                });

Persistência da última cidade: 

            
                const LAST_CITY_KEY = "lastCity";

                localStorage.setItem(LAST_CITY_KEY, city);

                const lastCity = localStorage.getItem(LAST_CITY_KEY);
                if (lastCity) {
                    DOM.cityInput.value = lastCity;
                    getWeather();
                }

Service Worker

No nível estrutural, o Service Worker foi corrigido com caminhos absolutos e registro adequado via servidor HTTP. Isso eliminou falhas silenciosas no cache.addAll() e elevou o projeto de uma PWA que "parece funcionar" para uma PWA consistente e confiável. 

            
                cache.addAll([
                    "/",
                    "/index.html",
                    "/style.css",
                    "/js/script.js",
                    "/manifest.json",
                    "/js/vendor/lottie-player.js"
                ]);

Remoção de dependência CDN

Por fim, foi removida a dependência de CDN do Lottie Player, adotando auto-hospedagem local. Isso resolveu bloqueios de storage causados por Tracking Prevention, melhorou o funcionamento offline e manteve o console limpo, o que demonstra uma postura de dono.

Solução:

  • Download do lottie-player
  • Hospedagem local (js/vendor)

Conclusão

O que mudou de verdade não foi a aparência, mas a base. O projeto deixou de ser apenas funcional e passou a ser evolutivo. O código tornou-se mais previsível, as responsabilidades ficaram claras, a experiência do usuário melhorou, o estado passou a ser persistido e a PWA ganhou consistência real.

Além da reorganização estrutural, o console ficou limpo, o cache passou a funcionar corretamente offline e o fluxo de erro tornou-se previsível, reduzindo comportamentos inesperados e aumentando a confiabilidade da aplicação.

O principal aprendizado foi entender que mais importante do que funcionar é ser fácil de entender, evoluir e manter. Código é um sistema vivo. Produto não é apenas interface. Produto é fluxo. Produto é previsibilidade. Produto é confiança para mudar.

Mentalidade que ficou: código bom não é o que funciona hoje, é o que continua funcionando quando você precisa evoluir amanhã.
